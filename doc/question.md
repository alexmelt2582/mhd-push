# 项目问答梳理

## 1. 从上游调用 send 到真正发送，完整链路是什么？

当前主链路可以拆成 6 步：
1. 先准备渠道账号，例如邮件账号、飞书机器人账号，统一通过 /account/save 创建。
2. 再准备模板，通过 /messageTemplate/add 创建模板、渠道、发送账号和模板内容。
3. 上游调用 /send 或 /batchSend，只传模板 ID、接收者、变量、业务扩展参数。
4. Web 层责任链完成前置校验、模板组装、接收者过滤，然后组装 TaskInfo。
5. SendMqAction 把 TaskInfo 列表投递到普通 Topic 或顺序 Topic。
6. Handler 消费后，真正调用第三方渠道发送。

关键变化：
1. 现在对外唯一的链路追踪 ID 是 messageId。
2. bizId 不再当作统一追踪 ID，只作为业务关联键或顺序键来源。

## 2. 调用 send 接口时，调用方到底应该传什么？

核心参数：
1. code：固定传 send。
2. messageTemplateId：模板 ID。
3. idempotencyKey：建议生产必传，用于接口级幂等。
4. messageParam.receiver：接收者，多个可逗号分隔。
5. messageParam.variables：模板变量。
6. messageParam.bizId：业务关联键，例如订单号、账单号、营销批次号。
7. messageParam.extra：扩展参数，例如 businessOwner、orderKey、dlqCallbackUrl。

明确一点：
1. 调用方不需要传 ContentModel。
2. 调用方也不应该把 bizId 当 messageId 用。
3. 真正用于后续 /trace/message、待确认查询、DLQ 查询的是接口返回的 messageId。

## 3. TaskInfo 里为什么以前会有这么多 ID，现在应该怎么理解？

现在应该按下面三个层次理解：
1. messageId：系统生成的唯一链路追踪 ID，从入口到 MQ、消费、发送、待确认、DLQ、trace 全程统一使用。
2. bizId：业务关联键，例如订单号，只用于业务归并、有序键来源、业务排障，不再作为统一 trace 主键。

当前已经做的收敛：
1. 发送接口回包只返回 messageId。
2. MessageParam 中的 bizId 注释已改为“业务关联键”。
3. TaskInfo 中明确了 messageId 才是统一 traceId。

## 4. Handler 层到底负责什么？

BaseHandler 现在承担的是发送模板方法，统一做 5 件事：
1. 先走 SendExecutionGuardService 做发送守卫，避免重复打第三方。
2. 再走 FlowControlFactory 做分布式限流和配额控制。
3. 再执行本地重试，而不是把所有失败都直接甩给 MQ 重投。
4. 发送成功后记成功状态，失败后记失败状态。
5. 对于“第三方可能已经收到了，但本地还没来得及确认”的情况，记录为 SEND_PENDING_CONFIRM。

这里要特别纠正一个旧理解：
1. 现在不是“只要 handler 失败就依赖 RocketMQ 重试”。
2. 当前主策略是 BaseHandler 内本地重试。
3. MQ 重试仍然存在，但不是所有发送异常的唯一兜底手段。

## 5. 如果第三方已经发成功了，但本地进程在确认前挂了，会怎样？

当前不会直接再去重复打第三方，而是进入待确认模型：
1. 发送前先在 Redis 写执行记录。
2. 如果后续判断这条消息可能已经发给第三方，但本地状态不确定，就标记为 PENDING_CONFIRM。
3. 后续重复消费或重复触发时，SendExecutionGuardService 会识别到这条消息是待确认，而不是继续重发第三方。
4. 运维侧可以通过待确认查询接口查看并人工确认成功或失败。

这就是现在的闭环：
1. 防止重复打第三方。
2. 保留人工确认入口。
3. 避免把“可能成功”误当“明确失败”。

## 6. 限流模块现在是什么实现，和旧逻辑有什么不同？

旧逻辑的问题：
1. 早期理解里更多是单机 RateLimiter。
2. 粒度偏粗，通常只到渠道级或账号级。
3. 不适合多实例共享额度。

当前 flowcontrol 的真实逻辑已经升级成分布式模式：
1. 统一由 FlowControlFactory 作为生产入口。
2. 配置从 flowControlRule 读取，支持按 CHANNEL、CHANNEL_ACCOUNT、CHANNEL_ACCOUNT_TENANT、ACCOUNT_ENDPOINT 四种粒度生效。
3. 底层依赖 Redis 做多实例共享限流状态。
4. 支持 qps、perMinute、perDay、429 backoff。
5. 支持 TOKEN_BUCKET 和 LEAKY_BUCKET 两种算法。

推荐使用原则：
1. 邮件、飞书、企微机器人这类通知型渠道，优先 TOKEN_BUCKET，因为允许一定突发但要控制长期速率。
2. 支付、结算、风控回调这类更强调匀速和稳定出流的通道，优先 LEAKY_BUCKET。

## 7. 为什么当前限流能适配多个邮箱账号、多个飞书 webhook？

因为 key 不是只按 sendChannel 做的，而是可以下钻到更细粒度：
1. CHANNEL：只按渠道限。
2. CHANNEL_ACCOUNT：按渠道 + 发送账号限。
3. CHANNEL_ACCOUNT_TENANT：按渠道 + 账号 + 业务方限。
4. ACCOUNT_ENDPOINT：按账号里的 webhook 或 endpoint 指纹限。

也就是说：
1. 两个邮箱账号的额度可以完全独立。
2. 两个飞书 webhook 的额度也可以独立。
3. 某个租户被限，不会把同渠道其他租户一起拖死。

## 8. 上游流量暴增时，系统现在怎么扛？

当前思路不是单点死扛，而是分层削峰：
1. Web 层先做接口幂等和参数过滤。
2. 模板内容在入 MQ 前就组装好，避免消费端重复做重活。
3. MQ 负责削峰填谷。
4. 消费端根据是否有序、业务方、线程池类型分开处理。
5. 真正调用第三方前再做细粒度限流。

结果是：
1. 上游爆量时不会直接把第三方瞬时打爆。
2. 系统可以允许 MQ 堆积，但不允许第三方通道被冲垮。

## 9. 顺序消息现在怎么保证顺序？

这里也要纠正一个旧理解：
1. 不是所有 RocketMQ 消费都走顺序消费。
2. 现在明确分成 unordered 和 orderly 两条路径。

当前策略：
1. 只有命中的 businessOwner 才会走顺序 Topic。
2. 顺序 Topic 里的消息会继续按 orderKey 落到本地分片单线程执行器。
3. 无序业务仍然走普通 Topic 和普通线程池。

这样做的目的：
1. 保证真正需要顺序的链路不被后续线程池打乱。
2. 又不会把所有业务都强行降成低吞吐顺序模式。

## 10. 消息重复发送现在怎么防？

现在至少有三层：
1. 接口级幂等：/send 和 /batchSend 支持 idempotencyKey。
2. 业务级去重：内容去重、频次去重还在。
3. 发送执行守卫：即使接口层漏掉，SendExecutionGuardService 也会阻止同一 messageId 再次重复打第三方。

所以真正的重复拦截不只在入口，而是入口、链路、第三方发送前各有一层。

## 11. 模板和变量的性能现在怎么做？

当前优化点主要在组装前移和模板缓存：
1. SendAssembleAction 负责在进入 MQ 前就把 ContentModel 组好。
2. 模板查询和结构解析走缓存，不是每次都查库和反射扫描。
3. 消费端拿到的是最终待发送内容，不需要再做模板拼装。

这意味着：
1. 消费端更轻。
2. 链路更稳定。
3. 压测时更能看出真正的发送瓶颈，而不是模板拼装瓶颈。

## 12. 线上排障时应该看什么 ID？

当前建议非常明确：
1. 查单条消息链路，看 messageId。
2. 查待确认，看 messageId。
3. 查 DLQ，看 messageId。
4. 查业务归并或某条订单链路，看 bizId。

## 13. 关于旧文档里哪些内容已经不准确？

下面这些旧说法现在都不准确，已经需要淘汰：
1. “限流还是单机 Guava RateLimiter”这个说法已经过期，现在生产入口是 Redis 分布式限流。
2. “handler 失败主要靠 MQ 重试”这个说法已经过期，现在优先是 BaseHandler 本地重试。
3. “bizId 也可以作为统一链路追踪 ID”这个说法已经过期，现在统一看 messageId。

### 线程池

#### 线程池的拒绝策略是什么？为什么使用这个？那不是会阻塞主线程吗？那此时其他的业务不是就不可以处理消息了吗？

当前业务线程池拒绝策略仍然是 `CallerRunsPolicy`，但现在在它前面补了一层“独立提交线程/分发队列”。

为什么这样做：
1. 这个策略不会直接丢任务。
2. 当业务线程池满了以后，由分发线程自己执行，相当于先把反压拦在分发层。
3. 这样热点渠道高峰时，优先变慢的是该渠道的分发线程，而不是直接拖住 MQ 消费线程。

关于“会不会阻塞主线程”：
1. 这里阻塞的不是 Web 主线程。
2. 现在 MQ 消费线程先把任务交给分发线程池，正常情况下不会再直接承担业务执行。
3. 只有当分发线程池自己的队列也被打满时，反压才会继续往前传，所以相比原实现，消费线程更不容易被热点渠道直接拖慢。

为什么仍然说不是“所有业务一起停掉”：
1. 当前任务真正执行的工作线程池依然是按 `groupId` 拆分的，也就是“渠道 + 消息类型”隔离。
2. 现在每个 `groupId` 还额外有一个独立分发线程池，所以热点链路的反压先落在自己的分发层。
3. 这意味着别的业务池和别的分发池通常不会被一起拖慢。

更准确的结论应该是：
1. `CallerRunsPolicy` 不会让任务丢失，但现在主要把执行压力反压到分发线程，而不是直接反压到消费线程。
2. 它的好处是保守、可靠；坏处是热点渠道高峰时会拉长该渠道分发层和业务层的处理时延。
3. 所以线上不能只靠拒绝策略，还要结合线程池容量、消费并发度、限流和 MQ 堆积监控一起看。

边界：
1. 如果某个热点渠道持续高峰，分发线程池和业务线程池仍然会变慢。
2. 如果这种情况持续，最终体现出来依然是消费吞吐下降、MQ 积压上升，而不是任务被丢掉。
3. 真正的治理手段应该是动态线程池调参、按渠道限流、必要时继续细分消费组或拆分热点链路。

验证方式：
1. 可以直接运行 `scripts/python_scenarios/scenario_s14_hotspot_acceptance.py`，它会真实并发调用当前项目的 `/send`。
2. 重点看 `avg_ms`、`p95_ms` 和 `success`，这些指标反映的是热点 burst 下接口受理是否稳定。
3. 脚本还会输出 `sample_message_ids`，可以继续结合 `/trace/message` 观察这些请求进入异步链路后的处理情况。

### 消息发送

#### 如果有多个业务，其中某些业务需要保持消息顺序，发送，如何保证这个业务消息发送时消息有序？

可以做到，系统现在是“按业务方选择性有序”，不是全局有序。

实现思路：
1. 发送端根据业务方标识（businessOwner）判断是否需要有序：命中有序白名单才走有序通道。
2. 有序业务消息会路由到专用有序 Topic，并携带稳定顺序键（优先取业务传入 orderKey，否则按 businessOwner + bizId 生成），保证同业务链路跨渠道仍在同一顺序键内。
3. 非有序业务走普通 Topic，不受有序消费约束，保证总体吞吐。

针对“购物车 -> 下单 -> 发货 -> 收货”且跨短信/邮件场景：
1. 这些节点使用同一个业务方标识和同一个 bizId（例如订单号），即可保证这条业务链路在消息系统内保持发送顺序。
2. 渠道可以不同，但顺序键一致时，系统仍按同一业务链路顺序投递。

边界与建议：
1. 顺序是“同一个顺序键内有序”，不是全局有序。
2. 要求上游对同一业务链路持续传同一个 bizId；否则无法建立稳定顺序关系。
3. 如果业务方需要更细粒度控制，可显式传入 orderKey。

关于资源消耗与优化：
1. 非有序业务不走 ORDERLY 消费，不会被有序消费额外开销影响。
2. 只有命中有序业务白名单才进入有序 Topic + ORDERLY 消费，资源消耗被限制在有序业务范围内。

#### 如何保证消息是否被消费了？

可以做到，系统现在提供“消费状态打点 + 自动重试 + DLQ记录”。

实现思路：
1. 消费成功会记录“消费成功”状态；消费异常会记录“消费失败”状态。
2. 消费异常会抛回 MQ，触发重试机制。
3. 达到最大重试阈值后，会落一份 DLQ 记录到 Redis，用于内部告警和人工补偿。

边界与建议：
1. “已消费”指进入系统消费流程并执行，不等于第三方渠道一定下发成功。
2. 需要配合日志检索、DLQ 列表和补偿接口查看最终处理结果。

#### 消息有重试机制吗？重试多少次？消息有兜底机制吗？

现在支持两层重试，并提供 DLQ 内部告警与人工补偿兜底。

实现思路：
1. 发送到 MQ 时支持应用层重试，默认最多 3 次，间隔 200ms。
2. MQ 消费失败自动重试，默认最多 3 次。
3. 超过消费重试阈值后，系统记录 DLQ 明细（messageId、payload、失败原因、重试次数等），支持后续内部告警和人工补偿。

DLQ内部告警与人工补偿接口：
1. DLQ 列表查询：分页查看待处理失败消息。
2. DLQ 详情查询：按 messageId 查看失败上下文。
3. 人工补偿：按 messageId 重新投递到原始 Topic（保留顺序键和标签）。
4. 内部告警：当消息最终失败时，系统向运维/值班组发邮件告警，提示人工处理。

关于“RocketMQ原生死信队列能不能用”：
1. 可以用，而且应该用。RocketMQ原生DLQ负责可靠兜底，不建议替代。
2. 在生产上，完全可以直接从 RocketMQ 原生 DLQ 中拉取失败消息，由运维/值班人员人工处理或重新投递。
3. 当前项目保留“应用侧DLQ记录”，不是因为原生DLQ不能用，而是为了补充更友好的检索、内部告警、补偿接口和失败上下文字段。
4. 更准确地说，原生DLQ是失败消息的事实来源，应用侧记录是便于排障和人工处理的增强层。
5. 如果团队后续更倾向于完全依赖 RocketMQ 原生 DLQ，也可以逐步把补偿入口切换到原生DLQ消费方案。

关于“ORDERLY消费者是否需要maxReconsumeTimes参数”：
1. 需要，建议显式配置，避免依赖默认值导致不同环境行为不一致。
2. 普通消费者与有序消费者都应设置相同阈值，保证重试策略一致。
3. 当前实现已为两类消费者都显式设置重试上限。

边界与建议：
1. 重试会提升成功率，但会增加延迟与流量峰值。
2. 业务侧必须保证幂等（按 bizId/messageId 去重），避免重试带来重复下发。

#### 如果用户消息推送一个20G的文件，你们是怎么处理的？有这个判断机制吗？mq我看有最大消息发送限制，那你们是怎么处理的？

20G 文件不能直接进 MQ，系统已增加硬性拦截机制。

实现思路：
1. 发送前会检查消息体大小（默认上限 3MB，低于 RocketMQ 默认 4MB，预留协议头开销）。
2. 超限请求会被直接拒绝，返回明确错误，提示将大文件上传到对象存储后只传 URL。
3. 系统消息中只承载“元数据 + URL”，不承载大文件本体。

边界与建议：
1. 大文件正确方案是“对象存储（OSS/COS/S3）+ 签名 URL + 过期控制”。
2. 若后续要做全自动上传，可在发送前增加文件中转服务，把文件自动上传后再替换为 URL。
