### 项目整体

#### 讲一下从上游到系统是怎么处理的，以及怎么发送的？

当前完整链路可以拆成四步：
1. 先准备发送源，也就是渠道账号。例如邮件账号、飞书机器人账号、短信账号，统一通过 `/account/save` 创建。
2. 再准备消息模板，也就是这个渠道要发什么格式、用哪个发送账号发、模板内容是什么，统一通过 `/messageTemplate/add` 创建。
3. 上游调用 `/send` 或 `/batchSend`，只传模板ID、接收者、动态变量和业务扩展参数。
4. 系统在发送责任链里查模板、组装 `ContentModel`、做参数校验后，再把 `TaskInfo` 投递到 MQ，最后由消费端真正调用第三方渠道发送。

#### 上游调用系统是怎么调用的，需要传递哪些参数？

如果是单条发送，当前接口是 `POST /send`。

核心参数：
1. `code`：固定传 `send`。
2. `messageTemplateId`：要使用的模板ID。
3. `idempotencyKey`：可选，但生产建议传，避免重复提交。
4. `messageParam.bizId`：业务流水号，建议传，用于链路追踪和有序投递。
5. `messageParam.receiver`：接收者。
6. `messageParam.variables`：模板占位符变量。
7. `messageParam.extra`：扩展参数，例如 `businessOwner`、`orderKey`。

如果是批量多条文案发送，当前接口是 `POST /batchSend`，差异主要在于传 `messageParamList`。

关键点：
1. 调用方不直接传 `ContentModel`。
2. 调用方只负责传模板ID和变量。
3. `ContentModel` 在系统内部由 `SendAssembleAction` 基于模板和变量组装出来。

#### 如何保证消息的全链路可靠？如果其中一个环节出现了问题，用户如何感知？如果保证消息不会丢失？

当前保障思路是“入口受理成功、MQ 持久化、消费重试、失败兜底、链路日志可追”。

具体做法：
1. 上游请求通过责任链完成参数校验、模板组装、去重等前置处理后，才会进入 MQ。
2. 发送到 MQ 失败时，接口会直接返回失败，不会给上游一个“假成功”。
3. MQ 收到消息后，消费端真正发送失败会触发 MQ 重试；达到阈值后落 DLQ 记录，便于内部告警和人工补偿。
4. 链路关键节点会打印结构化日志，并把消息状态日志、埋点日志投递到日志 Topic，方便后续统一消费和检索。

用户如何感知：
1. 同步阶段失败：上游接口立即拿到失败响应。
2. 异步阶段失败：业务侧通常通过消息状态查询、内部告警、补偿结果或业务回查感知，而不是靠一次同步接口完全覆盖。

为什么不会轻易丢：
1. 没进 MQ 的失败会同步报错，不会静默吞掉。
2. 进了 MQ 的消息有 Broker 持久化和消费重试兜底。
3. 超过重试上限后还有 DLQ 记录，不会“失败后无痕”。

#### 目前我看消息的全链路日志信息是保存在redis吗为什么不用mysql或者是否可以有别的工具可以替代，怎么保证消息不会丢失。

当前不是把“全链路日志”主存放在 Redis。

当前落点分三类：
1. 结构化运行日志直接打到本地日志文件。
2. 消息状态日志和埋点日志会同步发到日志 Topic，适合再接日志平台、数仓或检索系统。
3. Redis 当前更多承担的是幂等结果、DLQ 记录、敏感词字典这类“运行态数据”，不是长期日志仓库。

为什么不建议把全链路日志主存放到 Redis：
1. Redis 更适合低延迟缓存和短期状态，不适合长期海量日志留存。
2. 日志查询通常需要多维检索、聚合和长期保留，Redis 成本高且能力不匹配。

生产里更常见的日志落地方案：
1. MySQL：适合保存核心状态表、补偿表、审计表，但不适合承接超大规模原始日志明细。
2. Elasticsearch / OpenSearch：适合全文检索、链路排障、按 messageId 或 bizId 检索。
3. ClickHouse / Doris / Hive：适合做大量离线分析、报表和数仓。
4. Kafka / RocketMQ 日志 Topic + 下游消费落库：适合把“采集”和“存储”解耦。

如果只看“怎么保证消息不会丢失”，核心不在日志选型，而在消息主链路：
1. MQ 持久化。
2. 发送失败即时返回。
3. 消费失败自动重试。
4. 达到阈值后 DLQ 记录 + 人工补偿。
5. 关键节点日志可追踪，保证失败时能定位而不是无痕丢失。

#### 如何防止消息重复发送？

目前采用两层防重：
1. 接口级幂等：`/send` 和 `/batchSend` 支持幂等键，服务端使用 Redis 做短锁 + 结果缓存；同一幂等键的重复请求不会重复入队。
2. 业务级去重：发送链路里仍保留原有去重动作（内容去重、频次去重），防止渠道侧重复触达。

实现思路：
1. 第一次请求进入时，写入幂等锁并执行业务；完成后把响应缓存到 Redis。
2. 窗口期内重复请求直接返回缓存结果，不再重复发送。
3. 如果同一幂等键请求正在处理中，返回“处理中”状态，避免并发击穿。

业务级去重当前实现：
1. 内容去重：同一模板、同一接收者、同一内容，在指定时间窗口内只允许发送一次。
2. 频次去重：同一接收者、同一渠道，在当天内只允许发送 N 次。
3. 代码上现在已经拆成“规则构建（Builder）+ 执行过滤（Service）”两段，便于看清配置读取、参数组装和接收者过滤这三个步骤。

### 模板参数

#### 模板的动态参数是怎么处理的？怎么拼接的？

当前不是让调用方直接传 `ContentModel`，而是调用方只传：
1. `messageTemplateId`：告诉系统使用哪个模板。
2. `messageParam.variables`：告诉系统模板里的占位符应该替换成什么值。
3. `messageParam.receiver`、`bizId`、`extra`：接收者和业务上下文。

真正的拼接时机在发送责任链的 `SendAssembleAction`：
1. 先根据 `messageTemplateId` 从 `message_template` 表查到模板。
2. 根据模板里的 `sendChannel` 找到对应的 `ContentModel` 类型。
3. 读取模板中的 `msgContent` JSON。
4. 通过 `ContentHolderUtil.replacePlaceHolder` 用 `variables` 替换 `{$var}` 占位符。
5. 再通过反射把替换后的值填充到对应的 `ContentModel` 字段里。

也就是说：
1. 调用方不直接传 `ContentModel`。
2. `ContentModel` 是系统在 `SendAssembleAction` 阶段，基于“模板 + variables”组装出来的。
3. 组装完成后放入 `TaskInfo.contentModel`，再进入后续校验、MQ投递和消费发送流程。

#### 在使用模板和动态参数构建内容的时候比较消耗性能，你们是怎么处理的？怎么进行优化

如果一天发 1000 条消息，模板当然也会组装 1000 次，真正要优化的不是“单条消息只组装一次”这种事实，而是“把每次组装里重复的那部分成本尽量消掉”。

当前生产级优化思路是三层：
1. 模板查询层：用“两级缓存”降低数据库压力。
2. 模板解析层：把模板 JSON 解析结果、`ContentModel` 类型、字段反射元数据缓存在本地，避免每次都重新解析和扫描字段。
3. 动态变量层：每条消息仍然要做一次占位符替换，这是业务必要成本，但这部分只做字符串替换，不再重复查库和重复做模板结构解析。

当前实现已经按这个方向处理：
1. 本地缓存保存“模板 + 已解析的 msgContent + ContentModel 字段信息”，用于高频快速组装。
2. Redis 做二级缓存，保存模板原文和版本号，减少多实例场景下对 MySQL 的直接访问。
3. 本地缓存不是无脑永久信任，而是定期校验 Redis 中的版本号；版本一致就继续使用，不一致就失效并重新加载。
4. 模板新增、修改、删除时会主动清理缓存，尽量缩短脏数据窗口。

为什么要做多级缓存：
1. 只上本地缓存，单机快，但多节点下一致性差。
2. 只上 Redis，能减 DB 压力，但每次组装仍要经过网络访问，本地热点性能不够极致。
3. 本地 + Redis 的组合，才能同时兼顾延迟、吞吐和数据库保护。

多级缓存下一致性怎么保证：
1. 写路径主动失效：模板更新后，先更新数据库，再清理 Redis 和本地缓存。
2. 读路径版本校验：本地缓存会周期性检查 Redis 中的模板版本号，发现版本变化就丢弃本地快照并重建。
3. 这样不会要求每次读取都访问 Redis，但也不会让本地缓存无限期陈旧。

边界说明：
1. 占位符替换本身仍然是一条消息一次，这是不可省的业务成本。
2. 真正被优化掉的是“重复查模板、重复 parse JSON、重复反射扫描字段”这些固定成本。
3. 如果后续模板量更大、节点更多，还可以继续上“模板变更广播”或配置中心事件通知，把本地失效从“被动校验”升级成“主动推送”。

验证方式：
1. 可以直接运行 `scripts/python_scenarios/scenario_s12_template_cache.py`，它会真实调用当前项目的 `/messageTemplate/update` 和 `/send`。
2. 重点看模板更新后的 `cold_ms`，以及连续热发送后的 `hot_avg_ms`、`hot_p95_ms`。
3. 说明文档在 `doc/validation-scenarios.md`，里面给了命令和指标解释。

#### 多个业务方有多个业务模板测试怎么处理？每次手动添加新的模板是不是太费劲了？例如每个业务方的模板可能有许多，每个模板的动态参数都有可能有许多，你们是怎么处理的？

真实发送链路里，一般分三步准备：
1. 先创建渠道账号，也就是“可发送消息的源”。例如邮件账号、飞书机器人账号、短信账号，统一走 `/account/save`。
2. 再创建消息模板，也就是“发什么格式、哪个渠道发、用哪个发送账号”，统一走 `/messageTemplate/add`。
3. 最后调用 `/send` 或 `/batchSend`，传模板ID、接收者、动态变量和业务扩展参数。

当前项目里的测试和演练建议也是按这个顺序：
1. 先准备渠道账号。
2. 再准备模板。
3. 再由调用方传 `messageTemplateId + variables + receiver` 发消息。

为了降低多业务方测试成本，我已经补了 Python 场景脚本：
1. 脚本会先检查模板是否存在。
2. 不存在就自动调用模板接口创建。
3. 然后再模拟真实调用方请求 `/send`。
4. 这样不用每次手工进页面一个一个建模板。

如果一个业务方模板非常多，推荐做法：
1. 按渠道沉淀一批基础模板。
2. 把动态字段统一约定成固定命名，例如 `event`、`bizId`、`ts`、`url`。
3. 测试脚本里只切换模板ID和变量值，避免每次从零造模板。

### 敏感词过滤

#### 敏感词过滤采用的算法是什么？为什么采用（前缀树）此算法？相比别的算法的优点？

当前实现采用的是 Trie（前缀树）。

原因：
1. 敏感词匹配本质是“多模式字符串匹配”，前缀树适合做从某个起点往后连续匹配。
2. 多个敏感词如果前缀相同，Trie 可以复用前缀节点，避免重复比较。
3. 对比逐词 `contains` 扫描，Trie 更适合敏感词数量较多的场景。

当前实现方式：
1. 先把敏感词集合构造成 Trie。
2. 再对正文逐字符扫描。
3. 一旦命中敏感词，就按原长度替换成 `*`。

#### 敏感词更新后，如何保证服务节点及时生效？是重启还是热加载？

当前实现是热加载，不需要重启。

实现方式：
1. `SensitiveWordsConfig` 启动时先从指定文件路径读取词典。
2. 词典不会直接覆盖旧 key，而是先写入一个“带版本号的新词典 key”。
3. 新词典写完后，再切换 Redis 中的“当前版本指针”。
4. 各服务节点按当前版本拉取词典快照，并在本地缓存 Trie，版本变化时才重建。

#### 敏感词更新时，此时若有消息发送会不会产生敏感词无法匹配的问题？如何解决的？

你提到的“先删 Redis 再写 Redis，中间正好读到空集合”这个担心是对的，如果真这么做，确实可能让一条消息绕过敏感词过滤。

所以生产上不能用“删旧 key -> 回填新数据”这种覆盖式刷新，而要用“版本化快照切换”。

当前正确做法：
1. 先把新词典写到一个新的版本 key，例如 `dict:version:20260420xxxx`。
2. 确认新版本完整写入后，再把 `current_version` 指针切到新版本。
3. 读请求始终先拿当前版本，再去读对应版本的词典。
4. 老版本不会立即删除，而是保留一段 TTL，保证并发读请求即使读到旧版本，也还能拿到完整旧快照。

这样就不会出现“读到空词典”的窗口：
1. 指针切换前，读到的是完整旧版本。
2. 指针切换后，读到的是完整新版本。
3. 不会存在一个合法时刻让读路径拿到“版本存在但内容为空”的正常状态。

另外还做了一层本地保护：
1. 服务节点本地会缓存最近一次成功构建的 Trie 快照。
2. 如果 Redis 临时异常，或者新版本异常为空，节点会继续使用上一份有效快照，而不是直接按空词典放行。
3. 同一份词典内容会生成稳定版本号，不会因为定时刷新就不断制造新版本；如果只是 TTL 快过期，会续期当前版本，而不是切新版本。

验证方式：
1. 这一块更适合用“修改词典文件 + 观察服务日志 + 查看 Redis 当前版本 key”的方式验证。
2. 重点看相同内容重复刷新时是否仍然复用同一个 version。
3. 还要看版本集合 key 过期后，是否会回填同一个 version 并续期，而不是无意义地产生新版本。

#### 敏感词使用的是Redis中什么数据结构？是怎么处理的？如果有上万的敏感词怎么处理？

当前 Redis 中存的是 Set，但不是只存一个固定 key，而是“版本化 Set + 当前版本指针”。

处理方式：
1. `SensitiveWordsConfig` 把词典原始词条写入某个版本号对应的 Redis Set。
2. Redis 再保存一个 `current_version` 指针，表示当前该用哪一份词典。
3. 服务节点按版本读取一次词条集合，在本地构建 Trie 并缓存。
4. 只有版本变化时才重建 Trie，不会每条消息都重建。

如果敏感词上万：
1. Redis Set 存原始词条本身是可以承载的。
2. 真正的性能瓶颈不在 Redis 存储，而在“每条消息都拉全量词典并重建 Trie”。
3. 所以必须改成“Trie 本地缓存 + 版本变化时重建”，这样敏感词规模上来后 CPU 仍然可控。

### 限流

#### 我看限流部分是怎么实现的？在分布式下应该怎么实现？

当前实现是单机本地限流，核心依赖 Guava `RateLimiter`。

两种限流策略：
1. 请求级限流：一次请求按 1 个令牌获取。
2. 接收人数限流：按 `receiver.size()` 获取多个令牌。

在分布式下，如果要求“全局严格限流”，当前实现还不够，需要升级成：
1. Redis + Lua 的分布式令牌桶。
2. 或网关层统一限流。
3. 或按渠道在中间件层做集中式配额控制。

#### 项目中如果有一个上游消息暴增，因为系统内部可能有限流等应该怎么处理？

1. 内容去重：同一模板、同一接收者、同一内容，在指定时间窗口内只允许发送一次。
2. 频次去重：同一接收者、同一渠道，在当天内只允许发送 N 次。
3. 代码上现在已经拆成“规则构建（Builder）+ 执行过滤（Service）”两段，便于看清配置读取、参数组装和接收者过滤这三个步骤。

当前思路是“先削峰，再限流，再异步处理”：
1. 上游请求先入 MQ，避免同步压垮下游渠道。
2. 消费端按渠道和消息类型拆线程池。

1. 先根据 `messageTemplateId` 从 `message_template` 表查到模板。
2. 根据模板里的 `sendChannel` 找到对应的 `ContentModel` 类型。
3. 读取模板中的 `msgContent` JSON。

也就是说：

2. `ContentModel` 是系统在 `SendAssembleAction` 阶段，基于“模板 + variables”组装出来的。
3. 组装完成后放入 `TaskInfo.contentModel`，再进入后续校验、MQ投递和消费发送流程。
3. 真正调用第三方前再做限流，保护外部渠道。
4. 如果峰值持续过高，MQ 会形成堆积，但不会直接把第三方接口瞬时打爆。

当前实现的性能特点：

### 线程池

#### 线程池的拒绝策略是什么？为什么使用这个？那不是会阻塞主线程吗？那此时其他的业务不是就不可以处理消息了吗？

当前业务线程池拒绝策略仍然是 `CallerRunsPolicy`，但现在在它前面补了一层“独立提交线程/分发队列”。

为什么这样做：
1. 这个策略不会直接丢任务。
2. 当业务线程池满了以后，由分发线程自己执行，相当于先把反压拦在分发层。
3. 这样热点渠道高峰时，优先变慢的是该渠道的分发线程，而不是直接拖住 MQ 消费线程。

关于“会不会阻塞主线程”：
1. 这里阻塞的不是 Web 主线程。
2. 现在 MQ 消费线程先把任务交给分发线程池，正常情况下不会再直接承担业务执行。
3. 只有当分发线程池自己的队列也被打满时，反压才会继续往前传，所以相比原实现，消费线程更不容易被热点渠道直接拖慢。

为什么仍然说不是“所有业务一起停掉”：
1. 当前任务真正执行的工作线程池依然是按 `groupId` 拆分的，也就是“渠道 + 消息类型”隔离。
2. 现在每个 `groupId` 还额外有一个独立分发线程池，所以热点链路的反压先落在自己的分发层。
3. 这意味着别的业务池和别的分发池通常不会被一起拖慢。

更准确的结论应该是：
1. `CallerRunsPolicy` 不会让任务丢失，但现在主要把执行压力反压到分发线程，而不是直接反压到消费线程。
2. 它的好处是保守、可靠；坏处是热点渠道高峰时会拉长该渠道分发层和业务层的处理时延。
3. 所以线上不能只靠拒绝策略，还要结合线程池容量、消费并发度、限流和 MQ 堆积监控一起看。

边界：
1. 如果某个热点渠道持续高峰，分发线程池和业务线程池仍然会变慢。
2. 如果这种情况持续，最终体现出来依然是消费吞吐下降、MQ 积压上升，而不是任务被丢掉。
3. 真正的治理手段应该是动态线程池调参、按渠道限流、必要时继续细分消费组或拆分热点链路。

验证方式：
1. 可以直接运行 `scripts/python_scenarios/scenario_s14_hotspot_acceptance.py`，它会真实并发调用当前项目的 `/send`。
2. 重点看 `avg_ms`、`p95_ms` 和 `success`，这些指标反映的是热点 burst 下接口受理是否稳定。
3. 脚本还会输出 `sample_message_ids`，可以继续结合 `/trace/message` 观察这些请求进入异步链路后的处理情况。

### 消息发送

#### 如果有多个业务，其中某些业务需要保持消息顺序，发送，如何保证这个业务消息发送时消息有序？

可以做到，系统现在是“按业务方选择性有序”，不是全局有序。

实现思路：
1. 发送端根据业务方标识（businessOwner）判断是否需要有序：命中有序白名单才走有序通道。
2. 有序业务消息会路由到专用有序 Topic，并携带稳定顺序键（优先取业务传入 orderKey，否则按 businessOwner + bizId 生成），保证同业务链路跨渠道仍在同一顺序键内。
3. 非有序业务走普通 Topic，不受有序消费约束，保证总体吞吐。

针对“购物车 -> 下单 -> 发货 -> 收货”且跨短信/邮件场景：
1. 这些节点使用同一个业务方标识和同一个 bizId（例如订单号），即可保证这条业务链路在消息系统内保持发送顺序。
2. 渠道可以不同，但顺序键一致时，系统仍按同一业务链路顺序投递。

边界与建议：
1. 顺序是“同一个顺序键内有序”，不是全局有序。
2. 要求上游对同一业务链路持续传同一个 bizId；否则无法建立稳定顺序关系。
3. 如果业务方需要更细粒度控制，可显式传入 orderKey。

关于资源消耗与优化：
1. 非有序业务不走 ORDERLY 消费，不会被有序消费额外开销影响。
2. 只有命中有序业务白名单才进入有序 Topic + ORDERLY 消费，资源消耗被限制在有序业务范围内。

#### 如何保证消息是否被消费了？

可以做到，系统现在提供“消费状态打点 + 自动重试 + DLQ记录”。

实现思路：
1. 消费成功会记录“消费成功”状态；消费异常会记录“消费失败”状态。
2. 消费异常会抛回 MQ，触发重试机制。
3. 达到最大重试阈值后，会落一份 DLQ 记录到 Redis，用于内部告警和人工补偿。

边界与建议：
1. “已消费”指进入系统消费流程并执行，不等于第三方渠道一定下发成功。
2. 需要配合日志检索、DLQ 列表和补偿接口查看最终处理结果。

#### 消息有重试机制吗？重试多少次？消息有兜底机制吗？

现在支持两层重试，并提供 DLQ 内部告警与人工补偿兜底。

实现思路：
1. 发送到 MQ 时支持应用层重试，默认最多 3 次，间隔 200ms。
2. MQ 消费失败自动重试，默认最多 3 次。
3. 超过消费重试阈值后，系统记录 DLQ 明细（messageId、payload、失败原因、重试次数等），支持后续内部告警和人工补偿。

DLQ内部告警与人工补偿接口：
1. DLQ 列表查询：分页查看待处理失败消息。
2. DLQ 详情查询：按 messageId 查看失败上下文。
3. 人工补偿：按 messageId 重新投递到原始 Topic（保留顺序键和标签）。
4. 内部告警：当消息最终失败时，系统向运维/值班组发邮件告警，提示人工处理。

关于“RocketMQ原生死信队列能不能用”：
1. 可以用，而且应该用。RocketMQ原生DLQ负责可靠兜底，不建议替代。
2. 在生产上，完全可以直接从 RocketMQ 原生 DLQ 中拉取失败消息，由运维/值班人员人工处理或重新投递。
3. 当前项目保留“应用侧DLQ记录”，不是因为原生DLQ不能用，而是为了补充更友好的检索、内部告警、补偿接口和失败上下文字段。
4. 更准确地说，原生DLQ是失败消息的事实来源，应用侧记录是便于排障和人工处理的增强层。
5. 如果团队后续更倾向于完全依赖 RocketMQ 原生 DLQ，也可以逐步把补偿入口切换到原生DLQ消费方案。

关于“ORDERLY消费者是否需要maxReconsumeTimes参数”：
1. 需要，建议显式配置，避免依赖默认值导致不同环境行为不一致。
2. 普通消费者与有序消费者都应设置相同阈值，保证重试策略一致。
3. 当前实现已为两类消费者都显式设置重试上限。

边界与建议：
1. 重试会提升成功率，但会增加延迟与流量峰值。
2. 业务侧必须保证幂等（按 bizId/messageId 去重），避免重试带来重复下发。
