一、架构全景图：双端责任链模型
1. 发送端责任链 (Producer Chain)
   流程：API -> Chain Node 1 -> Node 2 -> Node 3 (MQ)
   Node 1: 校验节点 (ValidationNode)
   基础校验（非空、格式）。
   大文件预处理：检测 file 参数 -> 上传 OSS -> 替换为 url (解决 MQ 大小限制)。
   频控校验 (Redis Lua)。
   事务上下文初始化 (生成 traceId, shortCode)。
   Node 2: 组装节点 (AssemblyNode)
   解析 topic/to -> 计算接收人列表。
   动态模板渲染。
   关键动作：构建 MessageContext 对象，包含所有业务数据 + traceId。
   Node 3: 投递节点 (MqSendNode)
   核心逻辑：执行 RocketMQ 事务消息 发送。
   本地事务：落库 message_record (状态: PENDING)。
   MQ 事务：发送 Half Message -> Commit。
2. 消费端责任链 (Consumer Chain)
   流程：MQ Pull -> 路由分发 -> 专属线程池 -> Chain Node 1 -> Node 2 -> Node 3
   路由层 (Router)：
   读取消息头 channelType (email, feishu, sms...)。
   路由策略：根据 channelType 投递到对应的 独立线程池 (emailPool, feishuPool)。
   目的：防止短信接口超时拖死邮件发送。
   Node 1: 风控节点 (RiskControlNode)
   敏感词过滤 (DFA 算法)。
   黑名单校验。
   Node 2: 去重节点 (DeduplicationNode)
   Redis Bitmap/Set 校验 traceId 或 contentHash。
   防止网络抖动导致的重复消费。
   Node 3: 执行节点 (ExecutionNode)
   调用具体渠道 API (WeChat, SMS, Webhook)。
   更新 DB 状态 (SUCCESS/FAIL)。
   触发回调。
